# 다화자 대본 TTS 장기 아키텍처 및 편의 기능 구상

## 1. 목적

이 문서는 1차 구현에는 모두 넣지 않더라도, 프로젝트를 처음 만들 때부터 최종적으로 어디까지 확장할지를 잡아두기 위한 상위 설계 문서입니다.

핵심 방향:
- 로컬 전용
- GPU 활용 가능
- 긴 대본 작업에 강한 구조
- 줄 단위 편집과 재생성 지원
- 개인 제작 워크플로우에 최적화

## 2. 최종적으로 지향하는 제품 형태

최종 형태는 단순 생성 탭이 아니라 아래 세 가지가 합쳐진 로컬 스튜디오에 가깝습니다.

1. 대본 입력기
2. 줄별 렌더링 대시보드
3. 프로젝트/버전 관리기

즉, 사용자는 대본을 넣고 한 번에 끝내는 것뿐 아니라, 생성 후 결과를 들으면서 줄마다 다듬고 다시 만들 수 있어야 합니다.

## 3. 핵심 편의 기능 로드맵

### 3-1. 줄 단위 미리듣기

각 줄마다 즉시 재생할 수 있어야 합니다.

목적:
- 전체 합본을 다시 듣지 않아도 문제 줄을 바로 찾을 수 있음
- 빠른 수정 루프 가능

### 3-2. 줄 단위 다시 생성

마음에 안 드는 줄만 다시 생성합니다.

가능한 옵션:
- 같은 설정으로 다시 생성
- 속도/피치만 바꿔 다시 생성
- 다른 음성 팩으로 다시 생성

### 3-3. 대사 복사

줄 카드 또는 표에서 대사를 바로 복사합니다.

목적:
- 외부 메모장, 편집기, 자막 도구와 연동이 쉬움

### 3-4. 음성 파일 복사 또는 열기

로컬 전용 사용에서는 매우 유용한 기능입니다.

가능한 액션:
- 파일 경로 복사
- 포함 폴더 열기
- 파일 탐색기에서 선택

### 3-5. 버전 관리

한 줄을 여러 번 생성하면 버전을 남깁니다.

예:
- `001_v1`
- `001_v2`
- `001_v3`

사용자는 현재 버전 중 하나를 선택해 합본에 반영합니다.

### 3-6. 프로젝트 재개

앱을 껐다 켜도 이전 작업을 이어서 할 수 있어야 합니다.

저장 대상:
- 원본 대본
- 화자 매핑
- 줄별 현재 채택 버전
- 생성 이력
- 합본 상태

## 4. 로컬 전용 구조의 장점

이번 프로젝트는 개인용 로컬 프로그램이므로 아래 방식이 가능합니다.

- 모든 결과를 파일로 직접 관리
- 복사/이동/백업이 쉬움
- 인터넷 없이 사용 가능
- 민감한 음성을 외부 서버에 올리지 않아도 됨
- Windows 파일 탐색기와 자연스럽게 연결 가능

## 5. GPU 및 CUDA 활용 방향

GPU:
- NVIDIA GeForce RTX 5080급 사용 가능

활용 방향:
- 기본 생성은 CUDA 우선
- 같은 세션에서 반복 생성 시 모델 재로딩 최소화
- 줄 단위 재생성 시 필요한 자원만 재사용
- 향후 배치 생성 또는 선행 렌더링 지원

주의:
- 1차에서는 GPU가 충분해도 구조는 보수적으로 설계
- 먼저 정확한 상태 추적과 파일 구조를 안정화
- 이후 성능 최적화를 단계적으로 추가

## 6. 권장 데이터 구조

### 6-1. 프로젝트 단위 저장

```text
outputs/
└── multi_speaker/
    └── my_project/
        ├── project.json
        ├── script.txt
        ├── speakers.json
        ├── lines.json
        ├── renders/
        │   ├── line_001_v1.wav
        │   ├── line_001_v2.wav
        │   ├── line_002_v1.wav
        │   └── final_mix_v1.wav
        └── cache/
```

### 6-2. 줄 상태 예시

```json
{
  "line_index": 12,
  "speaker": "나레이터",
  "text": "오늘은 새로운 기능을 소개합니다.",
  "selected_version": "v2",
  "versions": [
    {"id": "v1", "path": "renders/line_012_v1.wav", "status": "ok"},
    {"id": "v2", "path": "renders/line_012_v2.wav", "status": "ok"}
  ]
}
```

## 7. 최종 대시보드 상상도

줄별 표 또는 카드에서 아래를 한 번에 다룰 수 있는 형태가 좋습니다.

- 줄 번호
- 화자
- 대사
- 현재 선택 버전
- 재생 버튼
- 다시 생성 버튼
- 대사 복사 버튼
- 경로 복사 버튼
- 합본 포함 여부
- 상태 표시

이 구조를 처음부터 염두에 두면, 1차에서 단순 생성만 구현해도 나중에 무리 없이 확장할 수 있습니다.

## 8. 1차 설계에 꼭 반영해야 할 미래 대비 포인트

1차에서 기능을 다 넣지 않더라도, 아래는 구조적으로 고려해야 합니다.

1. 줄마다 고유 ID를 둘 것
2. 결과 파일명을 줄 번호 기준으로 일관되게 저장할 것
3. manifest 또는 project 파일에 줄별 메타데이터를 남길 것
4. 줄별 재생성 버전을 나중에 추가할 수 있게 파일 구조를 열어둘 것
5. 화자 설정을 별도 구조체로 관리할 것
6. UI 이벤트를 줄 단위 작업으로 확장하기 쉽게 만들 것

## 9. 추천 구현 철학

추천 방향은 아래와 같습니다.

- 1차는 단순하게 만든다
- 하지만 데이터 구조는 프로젝트형으로 잡는다
- 줄 단위 식별자와 메타데이터를 남긴다
- 나중에 재생성, 버전관리, 복사 기능을 붙이기 쉽게 설계한다

즉, `기능은 작게`, `구조는 크게` 가져가는 방식이 좋습니다.
