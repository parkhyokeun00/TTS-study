# 다화자 대본 TTS 기능 설계안

## 1. 목표

현재 프로젝트의 TTS 기능은 주로 한 번에 한 화자 기준으로 음성을 생성합니다.
새 기능의 목표는 하나의 대본 안에서 여러 화자가 번갈아 말하도록 생성하고, 결과를 음성 파일로 저장하는 것입니다.

이 기능은 아직 구상 단계이며, 1차 목표는 안정적으로 동작하는 최소 기능 제품(MVP) 정의입니다.
동시에 최종적으로는 줄 단위 편집, 미리듣기, 부분 재생성, 프로젝트 저장, 대시보드 관리까지 확장 가능한 구조를 처음부터 염두에 둡니다.

## 1-1. 개발 전제

이 기능은 순수 개인용 로컬 프로그램으로 사용합니다.

설계 전제:
- 모든 처리와 저장은 로컬에서 수행
- 외부 서버 업로드 없음
- 생성 결과와 메타데이터는 로컬 파일로만 관리
- GPU 사용 가능
- CUDA 사용 허용

하드웨어 전제:
- NVIDIA GeForce RTX 5080급 GPU 사용 가능

설계 영향:
- 1차는 안정성 중심의 순차 처리로 시작
- 이후에는 캐시, 부분 재생성, 배치 생성, 선행 렌더링 같은 GPU 활용형 기능으로 확장 가능

## 2. 사용자 시나리오

대표 사용 시나리오는 아래와 같습니다.

1. 사용자가 긴 대본을 입력한다.
2. 대본 각 줄에는 화자 이름이 들어 있다.
3. 사용자는 각 화자 이름에 대해 사용할 음성 팩을 지정한다.
4. 프로그램은 줄 단위로 대사를 읽어 음성을 생성한다.
5. 생성된 줄별 음성을 저장한다.
6. 필요하면 전체를 하나의 합본 음성으로 저장한다.

예시 대본:

```text
나레이터: 오늘은 새로운 기능을 소개합니다.
여자1: 먼저 첫 번째 장면입니다.
남자1: 다음은 두 번째 설명입니다.
```

## 3. 핵심 개념

### 3-1. 화자

대본 안에서 대사를 읽는 역할 이름입니다.

예:
- `나레이터`
- `여자1`
- `남자1`

### 3-2. 음성 팩

화자에게 연결되는 음성 생성 설정 묶음입니다.
1차 버전에서는 저장된 `voice prompt(.pt)` 파일을 음성 팩의 핵심으로 사용합니다.

향후 확장 시 음성 팩에 아래 속성을 포함할 수 있습니다.

- `voice_prompt_path`
- `speed`
- `pitch_semitones`
- `ending_style`
- `ending_length_ms`
- `language`
- `volume_gain_db`
- `preferred_sample_rate`
- `silence_after_ms`
- `display_color`
- `tags`
- `notes`

### 3-3. 대본 항목

대본을 파싱한 뒤 내부적으로 관리하는 한 줄 단위 데이터입니다.

예상 구조:

```text
{
  "line_index": 1,
  "speaker": "나레이터",
  "text": "오늘은 새로운 기능을 소개합니다.",
  "voice_pack": "기본 나레이터",
  "status": "pending"
}
```

## 4. 범위 정의

### 4-1. 1차 구현 범위

- 텍스트 대본 입력
- `화자: 대사` 형식 파싱
- 화자 목록 자동 추출
- 화자별 음성 팩 지정
- 줄별 순차 TTS 생성
- 줄별 `.wav` 저장
- 전체 합본 `.wav` 저장
- 생성 로그 표시

### 4-2. 1차 구현에서 제외

- 음성 합성 중간 편집기
- BGM 자동 삽입
- 효과음 자동 삽입
- 화자별 감정 태그
- SSML 같은 고급 마크업
- 병렬 생성 최적화
- 실시간 재생형 스트리밍
- 줄별 버전 비교 UI
- 줄별 복사/내보내기 대시보드

### 4-3. 장기 확장 범위

초기 구현에는 넣지 않더라도, 최종적으로는 아래 수준까지 확장 가능하도록 구조를 열어둡니다.

- 줄별 미리듣기
- 마음에 들지 않는 줄만 다시 생성
- 줄별 여러 버전 저장
- 이전 버전과 현재 버전 비교 재생
- 줄별 대사 복사
- 줄별 음성 파일 경로 복사
- 줄별 음성 파일 열기
- 프로젝트 단위 저장/불러오기
- 화자 사전 관리
- 음성 팩 라이브러리 관리
- 긴 대본 작업 대시보드
- 생성 캐시 재사용
- GPU 배치 생성 최적화
- 생성 큐와 백그라운드 작업 관리

## 5. 입력 정의

### 5-1. 필수 입력

- 대본 원문
- 화자별 음성 팩 매핑
- 출력 파일명 또는 프로젝트 이름

### 5-2. 선택 입력

- 줄 사이 무음 길이(ms)
- 전체 합본 생성 여부
- 줄별 파일 저장 여부
- 출력 폴더명
- 언어 설정

## 6. 출력 정의

### 6-1. 파일 출력

예상 출력 위치:
- `outputs/multi_speaker/<job_id>/line_001_나레이터.wav`
- `outputs/multi_speaker/<job_id>/line_002_여자1.wav`
- `outputs/multi_speaker/<job_id>/final_mix.wav`
- `outputs/multi_speaker/<job_id>/script_manifest.json`

### 6-2. 화면 출력

- 총 줄 수
- 감지된 화자 수
- 현재 처리 중인 줄
- 성공/실패 여부
- 저장된 파일 경로

장기적으로는 아래 출력도 관리할 수 있어야 합니다.

- 줄별 오디오 플레이어
- 줄별 생성 버전 목록
- 줄별 재생성 버튼
- 줄별 복사 버튼
- 줄별 상태 배지
- 프로젝트 요약 카드

## 7. 대본 형식 제안

### 7-1. 기본 형식

가장 단순하고 초보자 친화적인 형식은 아래입니다.

```text
나레이터: 오늘은 새로운 기능을 소개합니다.
여자1: 안녕하세요. 첫 번째 장면입니다.
남자1: 다음 내용을 이어서 설명하겠습니다.
```

### 7-2. 규칙

- 각 줄은 `화자명: 대사` 구조를 따른다.
- 첫 번째 `:` 앞부분을 화자로 본다.
- 빈 줄은 무시한다.
- 화자명이 없는 줄은 오류로 표시하거나 이전 화자를 이어받는 방식 중 하나를 선택해야 한다.

1차 버전 추천:
- 화자명이 없는 줄은 오류로 처리

이유:
- 사용자가 수정 포인트를 바로 알 수 있음
- 자동 추정 로직보다 안정적임

## 8. 음성 팩 설계 제안

1차 버전에서는 별도 새 파일 포맷을 만들지 않고, UI 안에서 화자별 설정을 지정합니다.

예:

```text
나레이터 -> outputs/voice_prompt_narrator.pt
여자1 -> outputs/voice_prompt_female_01.pt
남자1 -> outputs/voice_prompt_male_01.pt
```

2차 버전부터는 `.json` 기반 음성 팩 설정 파일을 도입할 수 있습니다.

예상 구조:

```json
{
  "name": "기본 나레이터",
  "voice_prompt_path": "outputs/voice_prompt_narrator.pt",
  "speed": 1.0,
  "pitch_semitones": 0.0,
  "ending_style": "natural",
  "ending_length_ms": 180,
  "silence_after_ms": 120,
  "preferred_sample_rate": 24000,
  "tags": ["narration", "calm"]
}
```

## 9. 처리 방식 제안

### 9-1. 생성 전략

1차 버전은 순차 처리로 충분합니다.

이유:
- 디버깅이 쉽다
- GPU/메모리 사용량 예측이 쉽다
- 실패한 줄 추적이 쉽다

장기적으로는 GPU 여유가 충분하므로 다음 확장을 고려합니다.

- 다음 줄 선행 준비
- 이전 줄 캐시 유지
- 줄별 재생성 시 필요한 자원만 재사용
- 같은 음성 팩을 쓰는 줄 묶음 최적화
- 작업 큐 기반 백그라운드 생성

### 9-2. 합본 전략

- 줄별 생성 완료 후 순서대로 오디오를 연결
- 각 줄 사이에 설정된 무음을 삽입
- 마지막 합본 파일을 별도로 저장

## 10. 저장 정책

기존 프로젝트의 `outputs/` 정책을 유지합니다.

추천 구조:

```text
outputs/
└── multi_speaker/
    └── 20260316_153000_demo/
        ├── line_001_나레이터.wav
        ├── line_002_여자1.wav
        ├── line_003_남자1.wav
        ├── final_mix.wav
        └── script_manifest.json
```

`script_manifest.json`에는 아래 정보를 저장하는 것이 좋습니다.

- 원본 대본
- 파싱 결과
- 화자별 음성 팩 경로
- 생성 성공/실패 로그
- 최종 출력 파일 목록
- 줄별 생성 버전 정보
- 줄별 마지막 선택 버전
- UI 재개를 위한 프로젝트 상태

장기적으로는 작업 폴더를 프로젝트 단위로 다루는 것이 좋습니다.

추천 구조:

```text
outputs/
└── multi_speaker/
    └── project_demo/
        ├── project.json
        ├── script.txt
        ├── speakers.json
        ├── renders/
        │   ├── line_001_v1.wav
        │   ├── line_001_v2.wav
        │   ├── line_002_v1.wav
        │   └── final_mix_v1.wav
        └── cache/
```

## 11. 예외 처리 설계

아래 항목은 1차 버전부터 반드시 처리해야 합니다.

- 대본이 비어 있음
- `:` 없는 줄 존재
- 화자명은 있으나 대사가 비어 있음
- 화자에 연결된 음성 팩 없음
- 지정한 `voice prompt` 파일이 없음
- 모델이 아직 로드되지 않음
- 줄 생성 중 일부 실패

실패 정책 추천:

- 한 줄 실패 시 전체 작업을 중단할지 계속할지 옵션 제공
- 1차 기본값은 `실패 시 중단`

## 12. MVP 권장 결정

처음 구현할 때 아래처럼 단순화하는 것을 권장합니다.

- 대본 형식은 `화자: 대사`
- 음성 팩은 저장된 `voice prompt(.pt)` 파일 사용
- 후처리 값은 화자별 선택 옵션으로만 적용
- 생성은 순차 처리
- 출력은 `줄별 파일 + 전체 합본`
- 실패 시 즉시 중단
